# konfig-consumer

A thin, async **gRPC client** for the konfig service. It maintains a lock-free,
in-process view of one or more `Config`s by streaming the konfig `Subscribe`
RPC.

It does **not** watch the Kubernetes `Config` CRD directly — konfig the service
owns the kube watch + cache + fan-out. So this client needs:

- **no `kube::Client`**
- **no RBAC / Role / RoleBinding**
- **no ServiceAccount token** (no `automountServiceAccountToken` workaround)
- **no `kube` / `k8s-openapi` dependencies**

## API shape

```rust
use konfig_consumer::KonfigClient;

# async fn run() -> Result<(), Box<dyn std::error::Error>> {
// Built + spawned on the caller's runtime; no runtime is created here.
let client = KonfigClient::connect("http://konfig.svc:50051").await?;

// One Subscribe stream, two configs (multiplexed).
let handles = client.watch("default", &["risk-config", "limits"]).await?;

// Lock-free read; returns the default snapshot until the first event arrives.
let snap = handles[0].get();
let _max = snap.content["risk"]["max_order_size_usd"].as_u64();
# Ok(()) }
```

- `KonfigClient::connect(endpoint)` / `connect_tls(endpoint, ClientTlsConfig)` /
  `new(channel)` — you control the endpoint and TLS policy.
- `watch(namespace, &[names])` → one `Subscribe` stream covering N configs,
  returning one `ConfigHandle` per name (multiplexed: one client → N configs).
- `watch_one(namespace, name)` → single-config convenience.
- `ConfigHandle::get()` → `Arc<ConfigSnapshot>` via `ArcSwap::load_full`
  (atomic pointer load, no locks).

## Runtime model

The client **owns no tokio runtime**. `connect` builds its transport on the
caller's runtime, and `watch` spawns the stream-driver task with `tokio::spawn`
on the **ambient caller runtime**. The driver reads `ConfigEvent`s, demuxes by
`config.name`, reconnects with backoff (1, 2, 4, 8, 16, 30s cap), and resumes
from the last observed `resource_version`. Dropping the last `ConfigHandle`
from a `watch` call aborts that driver task.

## Consumption contract (READ THIS)

> Consume `konfig-consumer` as a **Cargo / crate_universe dependency in YOUR
> workspace** so it compiles against **your** `tonic` + `tokio` (one shared
> instance).
>
> Do **NOT** pull konfig in as a separate `bazel_dep` module for the client
> path — that builds a second `tonic`/`tokio` instance, and the stream driver's
> `tokio::spawn` then panics with `there is no reactor running`.
>
> You provide the endpoint (plus an optional mTLS `ClientTlsConfig`) and drive
> everything on your own runtime; the client owns no runtime and needs no kube
> access / RBAC.

## Build / test (Bazel only)

```sh
bazel test //rust/konfig-consumer:test //rust/konfig-consumer:fmt_test
bazel build //rust/konfig-consumer:clippy_test
```

Never `cargo test` directly. The proto client bindings are generated by
`build.rs` (`tonic_prost_build::configure().build_server(false)`) and wired
through the `konfig_consumer_bs` `cargo_build_script` in `BUILD.bazel`.
